# CLAUDE.md — StroyDocs Agent Directives

Проект: **StroyDocs** — SaaS для строительной документации (РФ, ГОСТ Р 70108-2025).
Стек: Next.js 14 App Router · TypeScript · Prisma · PostgreSQL · Timeweb S3 · shadcn/ui · Python/FastAPI (IfcOpenShell-сервис).
Архитектура: объект строительства как центр, URL `/objects/[objectId]/[module]/[tab]`.
Полный статус модулей — в `ROADMAP.md`. Детали стека — @docs/stack.md.

---

## 1. ОБЯЗАТЕЛЬНАЯ ВЕРИФИКАЦИЯ ПОСЛЕ КАЖДОЙ ПРАВКИ

После любой записи или редактирования файла ты ОБЯЗАН выполнить:

```bash
npx tsc --noEmit          # TypeScript — нет ошибок типов
npx eslint . --quiet      # ESLint — только ошибки, не предупреждения
```

**ЗАПРЕЩЕНО** сообщать «готово» пока оба скрипта не прошли без ошибок.
Если тайп-чекер не настроен — сообщи об этом явно вместо молчаливого пропуска.
Никогда не пиши «Done!» с висящими ошибками.

---

## 2. ЧТЕНИЕ ФАЙЛОВ ЧАНКАМИ

Файлы >500 строк читать ТОЛЬКО чанками — offset + limit параметры.
Один `read` на большой файл гарантирует неполное чтение и ошибки при редактировании.

```
Файл < 500 строк  → один read, можно доверять
Файл 500–2000 строк → читать чанками по 400–500 строк
Файл > 2000 строк → читать только нужный раздел, явно указывать offset
```

Tool results >50 000 символов обрезаются молча до 2000 байт — если поиск
вернул подозрительно мало результатов, перезапусти с более узким scope.

---

## 3. ПЕРЕД РЕФАКТОРИНГОМ — ОТДЕЛЬНЫЙ КОММИТ С ЧИСТКОЙ

Перед ЛЮБЫМ структурным рефакторингом файла >300 строк:

1. Удали мёртвый код: неиспользуемые props, exports, imports, debug-логи
2. Зафиксируй чистку отдельным коммитом: `git commit -m "chore: remove dead code in [filename]"`
3. Только после этого начинай рефакторинг

Это снижает контекстное давление и делает диффы понятными.

---

## 4. ПАРАЛЛЕЛЬНЫЕ SUB-AGENTS ДЛЯ ЗАДАЧ > 5 ФАЙЛОВ

Задача затрагивает >5 независимых файлов → ОБЯЗАТЕЛЬНО запускать sub-agents параллельно.
Каждый agent получает 5–8 файлов и свой контекстный window (~167K токенов).
Один agent на 20 файлов последовательно = гарантированный context decay.

Пример разбивки для модуля ПИР:
- Agent A: Prisma-схема + миграция + seed
- Agent B: API роуты (design-tasks, design-docs)
- Agent C: API роуты (pir-registries, pir-closure, analytics)
- Agent D: UI компоненты (список, карточки)
- Agent E: UI компоненты (аналитика, публичные страницы)

---

## 5. ПОСЛЕ 10+ СООБЩЕНИЙ — ПЕРЕЧИТЫВАТЬ ФАЙЛЫ

После 10 и более сообщений в сессии ОБЯЗАТЕЛЬНО перечитать файл перед правкой.
Авто-компакция могла молча уничтожить контекст — редактирование по памяти = сломанный код.

Признаки деградации контекста:
- Ты предлагаешь код который уже есть
- Ты «помнишь» функцию которой нет
- Правка не совпадает с реальным содержимым файла

При обнаружении → перечитай файл, сообщи: «Перечитываю после компакции».

---

## 6. БЕЗОПАСНОСТЬ ПРАВОК

**Перед каждым edit → перечитай файл.** Edit tool молча падает если old_string не совпадает.
**После каждого edit → перечитай файл** и убедись что изменение применилось.
Не делай более 3 правок одного файла без промежуточного чтения.

При переименовании функции/типа/переменной искать ВСЁ:
- Прямые вызовы и references
- Тайп-лэвел references (interfaces, generics)
- String literals содержащие имя
- Dynamic imports и re-exports
- Тестовые файлы и моки
- Barrel файлы (index.ts)

Grep не гарантирует полноту — assume it missed something.

---

## 7. ФАЗНОЕ ВЫПОЛНЕНИЕ

Многофайловые задачи — только фазами. Максимум 5 файлов на фазу.
Выполнить фазу → верификация (tsc + eslint) → ждать подтверждения → следующая фаза.

Когда пользователь говорит «сделай план» — только план, без кода.
Когда пользователь говорит «да», «делай», «го» — выполняй без повторения плана.

---

## 8. МЁРТВЫЙ КОД И ЧИСТОТА

После любой реструктуризации: удали всё теперь неиспользуемое.
Если файл >150 строк компонента — предложи разбить.
Если файл >300 строк — обязательно разбить (правило проекта).

Разделение UI и логики — ВСЕГДА:
- `ComponentName.tsx` — только JSX (~60–80 строк)
- `useComponentName.ts` — запросы, стейт, бизнес-логика

---

## 9. MULTI-TENANCY — ПРОВЕРЯТЬ В КАЖДОМ API РОУТЕ

КАЖДЫЙ новый API роут обязан начинаться с:

```typescript
const session = await getSessionOrThrow();
// затем проверка принадлежности к организации:
const entity = await db.someModel.findFirst({
  where: { id: params.id, organizationId: session.user.organizationId }
});
if (!entity) return errorResponse('Не найдено', 404);
```

Никогда не фильтровать только по `id` без `organizationId` — это утечка данных между тенантами.

---

## 10. ПРАВИЛА КОДА ПРОЕКТА

- TypeScript strict mode — никаких `any`
- Комментарии к бизнес-логике — **на русском языке**
- Название проекта: **StroyDocs** (не stroydocs, не Stroydocs)
- Файлы только в Timeweb S3 — не использовать зарубежные CDN для ПДн (ФЗ-152)
- Пагинация для любого `findMany` на больших моделях: `take`/`skip`, лимит 50, макс 200
- Новые Prisma поля с индексом → `@@index([field])` + `npx prisma migrate dev`

---

## 11. САМОПРОВЕРКА ПЕРЕД «ГОТОВО»

Перед тем как сообщить о завершении задачи:
1. Перечитай все изменённые файлы
2. Убедись что ничего не ссылается на удалённое
3. Убедись что нет неиспользуемых imports
4. Запусти `npx tsc --noEmit` + `npx eslint . --quiet`
5. Опиши что именно проверил — не просто «выглядит хорошо»

При фиксе бага: объясни **почему он произошёл** и что можно сделать чтобы
предотвратить этот класс ошибок. Не просто исправляй — делай post-mortem.
